# Jellow Communicator Documentation Style Guide

## Overview

The Jellow Communicator Project style guide establishes consistent documentation standards for all project materials. Since Jellow is designed to support communication accessibility for children and individuals with speech and language difficulties, all documentation must be simple, supportive, inclusive, and easy to understand.

The documentation team maintains:

- A clear and user-friendly tone
- A professional but approachable writing style
- An accessibility-focused communication approach
- Simple and encouraging language for users, caregivers, teachers, and developers

---

## Voice Guidelines

### Writing Principles

- Write with empathy and clarity.
- Keep instructions short and direct.
- Avoid overly complicated words when unnecessary.
- Prioritize readability for beginners and non-technical users.

### Active vs. Passive Voice

- Use active voice whenever possible.
- Use passive voice only when the subject performing the action is unknown or not important.

---

## Terminology and Word Choice

Because Jellow Communicator is designed for children and individuals with speech and language difficulties, all terminology must remain simple, supportive, inclusive, and easy to understand.

The documentation should avoid complicated technical terms whenever simpler alternatives are available.

### Language Guidelines

- Use short and familiar words.
- Use positive and encouraging language.
- Avoid medical or highly technical terms unless necessary.
- Write in a way that caregivers, teachers, and beginners can easily understand.
- Prioritize clarity and accessibility over technical precision.

### Preferred Style

- “Tap the picture to speak.”
- “Choose an icon.”
- “Open Settings to change the voice.”
- “Create a communication board.”

### Avoid

- “Execute speech output.”
- “Configure interface parameters.”
- “Initialize AAC functionality.”
- “Deploy communication assets.”

### Preferred Examples

- “Children with speech difficulties”
- “Users who need communication support”
- “People using AAC tools”

### Avoid

- “Speech-impaired users”
- “Disabled child”
- “Non-verbal case”

### Tone Reminders

The documentation should sound:

- Friendly
- Calm
- Encouraging
- Clear
- Supportive

Avoid language that feels overly formal, robotic, or technical.

# Markdown Formatting Rules

## Bold Text

### Rule

Use bold text only for:

- User interface elements
- Buttons
- Menu options
- Screen labels
- Important warnings

Use bold formatting when referring to items users can directly see or tap in Jellow.

### Correct Examples

- Open **Profile** to edit caregiver information.
- Select **My Boards** to create a communication board.
- Tap **Save** to store changes.
- Choose **Settings** to customize preferences.

### Incorrect Example

```text
This feature is very useful.
```

### Correct Example

```text
This feature is very useful.
```

---

## Code Blocks and Backticks

### Rule

Use inline backticks for:

- File names
- Variables
- Commands
- Folder names
- Technical references

### Examples

- `settings.json`
- `userProfile`
- `boardData`

Use code blocks only when showing multiple commands or implementation examples.

Avoid using code formatting for interface labels.

### Incorrect

```text
Select `My Boards`
```

### Correct

```text
Select My Boards
```

---

## Heading Levels

### Rule

Maintain a consistent heading hierarchy.

### Use H2 for Major Sections

- Creating Boards
- Managing Profiles
- Customizing Settings

### Use H3 for Subsections

- Adding Icons
- Editing Caregiver Information

---

# Punctuation and Mechanics

The documentation for Jellow Communicator must follow consistent punctuation and writing mechanics to maintain clarity, readability, and accessibility for children, caregivers, teachers, and beginner users.

### Correct Example

- Open Settings to change the voice.

---

## Bullet List Standards

Use bullet lists for steps, features, and grouped information.

### Rules

- Start every bullet point with a capital letter.
- End complete sentences with a period.
- Do not use periods for short fragments or single-word items.
- Keep list formatting consistent within the same section.

### Correct Example

- Open Profile.
- Select Save.
- Return to the dashboard.

### Correct Fragment List

- Communication boards
- Voice settings
- User profiles

### Incorrect Example

- open Profile
- Select Save
- return to the dashboard

---

## Oxford Comma

The team standard is to use the Oxford comma for clarity and readability.

### Correct Example

The app supports children, caregivers, and teachers.

### Incorrect Example

The app supports children, caregivers and teachers.

---

## Headings

Use Sentence Case for all headings.

---

## Capitalization

- Capitalize interface elements exactly as they appear in the application.
- Avoid unnecessary capitalization in normal sentences.

---

## Quotation Marks

- Use quotation marks only when directly quoting text or referring to exact words.
- Avoid excessive quotation marks for emphasis.

---

## Exclamation Marks

Use exclamation marks sparingly.

---

## Ellipses

Avoid unnecessary ellipses (`...`) because they may confuse beginner readers.

### Correct Example

- Select Next to continue.

### Incorrect Example

- Select Next to continue...

---

## Numbers

- Spell out numbers one through nine in regular text.
- Use numerals for 10 and above.
- Use numerals for steps, measurements, versions, or technical references.

### Correct Examples

- Create three communication boards.
- The app supports 12 languages.
- Complete Step 2.

---

## Contractions

Contractions are allowed when they improve readability and make the tone feel more natural.

### Acceptable Examples

- You’ll
- Don’t
- It’s

Avoid excessive informal language or slang.

---

## Accessibility Reminder

Punctuation and mechanics should always support:

- Easy reading
- Clear instructions
- Consistent formatting
- Accessibility for beginner and non-technical users

---

# Do’s and Don’ts

| Do’s | Don’ts |
|---|---|
| Use simple and clear language | Use difficult technical words |
| Write short and direct instructions | Write long and confusing instructions |
| Use active voice | Use too much passive voice |
| Use “documentation” in formal writing | Use “docs” in documentation |
| Explain acronyms on first mention | Use unexplained acronyms immediately |

---

# Accessibility and Communication Goal

The overall goal of the Jellow Communicator documentation is to provide accessible, beginner-friendly, and supportive guidance for all users, especially children, caregivers, teachers, and individuals who rely on AAC tools for communication.
